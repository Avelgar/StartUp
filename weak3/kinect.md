# Проект «Kinect» — система трекинга тела для Windows

---

## 1. Описание сегмента (границы)

Мы рассматриваем сегмент **разработчиков и технических энтузиастов** (CV/AI, инди-игры, интерактив), поскольку они испытывают острую потребность в дешёвом body tracking без специализированного оборудования, готовы интегрировать ранние версии SDK в свои проекты и ценят локальную обработку данных.

## 3. Persona

### Илья, 24 года — инди-разработчик игр и интерактива

**Краткое описание:**
Илья разрабатывает небольшие игры и интерактивные прототипы на Unity для Windows. Он хочет добавить управление жестами, но покупать VR-трекеры или искать б/у Kinect для тестов — слишком дорого и долго. Готов читать документацию и экспериментировать с кодом, но не хочет тратить недели на сборку низкоуровневых CV-пайплайнов с нуля.

**Цели:**
- Быстро протестировать идею игры с управлением жестами (за 1-2 вечера)
- Интегрировать данные скелета в свой движок через понятный SDK
- Получить приемлемый FPS на среднем ноутбуке пользователя
- Избежать привязки игроков к покупке специфического оборудования

**Боли и фрустрации:**
- Аппаратные решения (Kinect/Azure) дорогие и снимаются с производства
- Открытые CV-модели (OpenPose) требуют мощных GPU и сложны в настройке
- Управление жестами часто получается некомфортным из-за задержек
- "Сырые" решения ломаются при плохом освещении или перекрытии рук

**Контекст работы:**
Работает за ПК на Windows, использует игровые движки. Создает приложения, которые должны запускаться локально у пользователей с разными веб-камерами и без интернета.

**Текущий процесс:**
Ищет открытые библиотеки на GitHub, пытается запустить демо. Если интеграция занимает больше пары дней или требует сложных зависимостей — отказывается от идеи и делает обычное управление мышью/клавиатурой.

**Критерии выбора (топ-5):**
1. Готовый SDK для Windows с понятными примерами
2. Работа с любой веб-камерой "из коробки"
3. Низкая задержка (realtime)
4. Поведение при плохом свете и окклюзиях
5. Понятная документация

**Возражения и риски:**
- «Будет ли это работать стабильно без depth-камеры?»
- «Не съест ли трекинг весь FPS моей игры?»
- «Что будет, если игрок повернётся боком?»
- «Насколько сложно это встроить в мой текущий проект?»
- «Будет ли это работать на старых веб-камерах?»

**Где «обитает»:**
GitHub, itch.io, Reddit (r/gamedev, r/computervision), Discord-сервера инди-разработчиков, Habr.

## 5. Инсайты и гипотезы

### Инсайты о сегменте

1. **Интеграция важнее абсолютной точности:** Для разработчиков готовый SDK с примерами ценнее, чем "голая" модель с точностью на 10% выше.
2. **Hardware — главный барьер:** Необходимость покупки сенсоров убивает конверсию инди-проектов на старте; "no-hardware" подход критичен.
3. **Задержка (Latency) определяет UX:** В жестовом управлении пользователь прощает неточность позы, но не прощает задержку отклика >100мс.
4. **Окклюзии — главная проблема CV:** Перекрытие частей тела (рука за спиной) — основная причина "поломки" UX, которую нужно решать на уровне SDK.
5. **Потребность в локальности:** Игры и фитнес-приложения должны работать автономно; облачные API для видеопотока неприемлемы из-за пинга и приватности.
6. **Демо продаёт продукт:** Разработчик принимает решение об использовании SDK за первые 15 минут запуска демо-приложения.
7. **Мультидоменность:** Одни и те же технические требования (FPS, локальность) нужны и gamedev, и фитнесу, и создателям интерактивных стендов.

### Гипотезы

| № | Гипотеза |
|---|---|
| H1 | Для разработчиков интерактивных приложений в сценарии прототипирования ключевая боль — дороговизна hardware-сенсоров — приводит к отказу от идеи, поэтому они готовы использовать webcam-based SDK при условии стабильного FPS |
| H2 | Для целевого сегмента сложность самостоятельной интеграции MediaPipe является барьером, поэтому они выберут готовый Windows-SDK (даже платный) вместо бесплатной сборки пайплайна с нуля |
| H3 | В сценарии управления ПК/играми задержка трекинга является критичным фактором: при задержке выше порога комфорта пользователи отказываются от продукта, даже если точность скелета 100% |
| H4 | Локальная обработка (без отправки видео в облако) является обязательным требованием для >80% разработчиков из соображений приватности и пинга |
| H5 | Наличие готовых демо-сценариев (например, Unity-префаб) повышает конверсию в установку SDK в 2-3 раза по сравнению с простой документацией API |
| H6 | Программное сглаживание окклюзий (предсказание положения перекрытых частей тела) в SDK является важным конкурентным преимуществом перед "сырыми" CV-моделями |
| H7 | Инди-разработчики готовы использовать бесплатную версию с водяным знаком, а коммерческие студии купят Pro-лицензию для white-label использования |

## 6. ТОП-3 гипотезы и план проверки

### H2 — Готовый SDK против самостоятельной интеграции

**Гипотеза:** Разработчики выберут готовый Windows-SDK вместо самостоятельной интеграции MediaPipe из-за экономии времени.

| Параметр | Описание |
|---|---|
| **Метод проверки** | Глубинные интервью (10-15 чел.) + Юзабилити-тест: дать задачу вывести скелет через "сырой" MediaPipe и через наш SDK (замерить время) |
| **Критерий подтверждения** | >70% респондентов подтверждают боль интеграции; в тесте интеграция через SDK занимает в 3+ раза меньше времени при меньшем количестве ошибок |

---

### H1 — No-hardware подход как решение барьера

**Гипотеза:** Пользователи отказываются от motion-проектов из-за необходимости сенсоров и перейдут на webcam-решение.

| Параметр | Описание |
|---|---|
| **Метод проверки** | Запуск Landing Page с демо "Kinect через веб-камеру" и кнопкой "Получить ранний доступ к SDK" (Fake Door test) на профильном трафике (Reddit, Habr) |
| **Критерий подтверждения** | Конверсия из посетителей лендинга в заявки на SDK ≥10%; в последующих опросах заявок >50% указывают цену/отсутствие сенсоров как главную проблему |

---

### H3 — Задержка как критический фактор UX

**Гипотеза:** Задержка отклика является более критичным фактором для жестового управления, чем абсолютная точность позы.

| Параметр | Описание |
|---|---|
| **Метод проверки** | A/B тест прототипа управления игрой: версия с искусственной задержкой (150мс) и высокой точностью VS версия с минимальной задержкой (30мс), но меньшей точностью |
| **Критерий подтверждения** | Участники теста оценивают "отзывчивость" и общее удовлетворение от версии с низкой задержкой на 2+ балла выше (по 10-балльной шкале), несмотря на артефакты точности |
